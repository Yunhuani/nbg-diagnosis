云华 NBG 五维分析 · Prompt 规范 V1.1

本规范是「五维分析编排」模块的内容层依据。它管的是「分析的内容怎么被生成出来」——五维各用什么提示词、怎么逼出有棱角的判断、怎么挖出真实的反转发现、怎么从机制上杜绝编造。
与其他文档的分工：

《NBG 诊断报告制作规范》管「报告长什么样」（视觉/语言/版式/结构）。
《采集输入需求清单》管「向客户收什么」（本规范的输入来自它）。
《NBG 产品决策备忘》管「产品怎么做」（技术路线/可信度红线，本规范是它在内容层的落地）。
本规范管「分析的内容怎么生成」。报告规范是「皮」，本规范是「脑」。

适用对象：诊断阶段的五维分析编排。方案阶段的 Prompt 规范另行制定。


一、一句话定位：把"裸聊 AI"和"NBG 诊断"区分开的全部秘密，在这一层
客户自己问豆包/ChatGPT，也能得到五段像模像样的分析。我们能收费、能建立信任，只因为这一层做了四件裸聊做不到的事：

逼出因果链——不停在"现象描述"，把一个数据顺着 So-What 追问三四层，逼出客户想不到的结论。
量化带参照系——每个数字配一条基准线，证明"我们见过同类企业"。
挖出反转发现——但只挖真的、经得起反问的；假反转比没有反转更致命。
交叉呼应——同一事实从五个维度交叉照射，形成单点提问给不出的全局洞察。


本规范的核心立场：上面四件事，没有一件能靠"在 prompt 里下命令"可靠实现。下命令只能得到"看起来像"的版本（编造的因果、套话的参照系、glib 的伪反转）。每一件都必须用机制保证——要么是结构化的流程约束，要么是可被校验的输出格式。本规范的主体，就是这些机制。


二、编排总架构（Codex 按此搭建管线）

方法论与决策备忘定的是「一维一次调用 + 最后交叉验证」。本规范在此基础上把它操作化为下面的管线。对外部数据维度（市场、竞争），单维拆成「检索 + 推理」两次调用，这是杜绝编造的机制要求，不是复杂化。

输入层
 ├─ 采集数据（结构化 JSON，来自《采集清单》schema）
 └─ 财务计算结果（确定性 JSON，来自财务模块）—— 视为 ground truth，全程不可改

Step 0 · 事实底座组装（无大模型调用，纯代码）
 └─ 把采集数据 + 财务结果 + 基础/Plus 可用性映射，拼成一个「共享事实底座」对象，
    所有维度调用都拿同一份。

Step 1 · 检索取数（仅市场、竞争两维需要）
 └─ 真实联网检索 → 产出「来源语料」（每条带 url + 信源层级）。
    见第四节。这是分析的唯一合法外部事实来源。

Step 2 · 五维分析（5 次独立调用，可并行）
 └─ 每维 = 通用系统提示词 + 该维专用提示词 + 共享事实底座（+ 该维的来源语料）。
    每维输出一份固定 JSON（含核心判断、因果链、证据、反转候选、评分、降级标记）。
    ★ 五维彼此独立、互不喂入对方结论——防止级联幻觉。跨维联系全部留到 Step 3 集中做。
    ★ 但五维都能看到同一份财务 ground truth（商业模式/竞争判断依赖毛利真相）。

Step 3 · 交叉验证与综合（1 次调用，用最强模型）
 └─ 接收全部五维 JSON → 找交叉呼应、检验并筛选反转、给总体判断、
    定五维评分一致性、产出「三个关键发现」、做一致性自检。见第七节。

输出层
 └─ 一份结构化诊断结论对象，喂给报告生成模块。
为什么五维独立、跨维联系集中到 Step 3：顺序喂入（财务结论喂给竞争维…）看似省事，实则会让前一维的错误判断/幻觉级联放大到后面。更稳的做法是五维各自只面对客观事实底座、保持独立，把所有"同一事实两种含义"的交叉照射，放到一个能同时看到五份输出、可被专门校验的综合环节里刻意地做。交叉呼应是护城河，必须当成一个独立的、最值钱的环节认真设计，而不是跑完五维后顺手做的最后 10%。
调用用途模型档位建议温度检索取数抓真实网页、抽事实带检索能力的模型低市场/竞争分析反转密集、依赖外部事实顶级模型0.2–0.4商业模式/内部能力/财务分析判断为主顶级或次级0.2–0.3综合验证全局洞察、反转筛选必须顶级模型0.2–0.3

温度统一压低：诊断要的是判断的一致性和可复现，不是文采的随机性。所有调用强制 JSON 输出。


三、三个跨维度的硬机制（写进每一维的提示词）
3.1 机制一：So-What 因果链强制（治"现象描述"病）
裸聊的通病是停在一层观察（"你客户集中度高"）。NBG 的判断必须是链条的终点。
做法：每一维提示词强制模型先内部产出一条 ≥3 环的因果链，再给结论；输出里保留结论 + 最锋利的 1–2 环。
在给出核心判断前，你必须先完成一条因果追问链（reasoning_chain），
每一环都回答上一环的"那又怎样（So-What）？"，至少追到第三层：

  事实 → 直接后果 → 它逼出的经营/财务后果 → 客户没意识到的结构性结论

示例（不要照抄，理解结构）：
  前三大客户占 65%（事实）
  → 定价权实际在客户手里，你只能接受压价（直接后果）
  → 越压价越靠走量守单，毛利被持续侵蚀（财务后果）
  → 你以为在"做生意"，实际在"用现金流给大客户做无息垫资"（结构性结论）

只有走完链条得到的结论，才允许写进 core_judgment。
若一条事实追三层后只能得到客户显然已知的结论，标记该发现 strength=low，
不要硬拗出"深刻"的样子。
3.2 机制二：量化必带参照系（治"裸数字"病）
report 里出现的每一个关键数字，必须配一条参照系（benchmark），否则不准单独出现。
参照系优先级：
  1) 行业健康线 / 同业典型值（来自检索语料，带源）——最有力
  2) 该企业自身的历史值（来自采集的趋势数据）
  3) 同类企业经验区间（明确标注为"经验判断"，不伪装成硬数据）
形式：「65% vs 行业健康线约 40%」「净利率 6% vs 该子品类典型 12–15%」。
找不到任何参照系的数字，宁可不写，或降级为定性表述。

这条直接对应制作规范"量化带参照系"。注意第 1 类参照系本身是外部事实，必须走第四节的检索机制取得，不能模型脑补一个"行业健康线 40%"。

3.3 机制三：反转发现的有效性检验（本规范最重要的独立约束）

这是我对现有标准做的最重要的加强。 制作规范要求"每维至少一个反转认知发现"，方向完全正确——反转是最有杀伤力的洞察。但若在 prompt 里把它写成硬命令（"必须产出一个反转"），模型会批量制造假反转：把任何风险都 glib 地反夸成资产。一个被精明老板一眼看穿的假反转，会像决策备忘里说的"看似精确实则编造的对手数字"一样，让整份报告的信任瞬间崩塌。所以反转不能只下命令，必须配一道可证伪的有效性检验。

一个反转发现（"你以为的风险其实是资产"）只有同时通过四关才允许进入报告：
关卡检验问题不通过则① 风险真实客户是否真的把这件事当成风险/软肋？不是真痛点 → 不是反转，是废话② 机制具体能否说出一个具体的、有依据的结构性原因，解释它为什么反而是资产？说不出机制只会反夸 → 删除③ 可证伪能否说清"在什么条件下这个反转就不成立"？说不出失效条件 → 是 denial（自我安慰），不是洞察④ 可决策它是否指向一个和"天真读法"不同的决策？不改变任何决策 → 没价值
写进竞争维与商业模式维提示词（反转最易出现在这两维）：
尝试寻找一个"反转认知"发现：客户视为风险/软肋、实则是难以复制资产的东西。
若找到，必须按下面结构输出，缺任何一项则判定为伪反转、置 reversal_candidate=null：
{
  "naive_reading": "客户/常识对这件事的天真读法（视为风险）",
  "reframe": "它其实是资产的反转判断",
  "mechanism": "为什么是资产的具体结构性机制（须可被检验，不是修辞）",
  "falsifier": "在什么条件下这个反转就不成立、反而是真风险",
  "confidence": 0.0-1.0,
  "status": "machine_confirmed | needs_human_falsifier_check",
  "depends_on": ["支撑此判断所需的关键输入，如：认证/独有能力清单"]
}
宁可置 null，绝不编造。低于 0.6 置信度的反转，交由综合环节决定是否采用。

★ status 的判定（关键）：失效条件（falsifier）往往依赖检索查不到、模型判不了的
私有/默会信息（如"该大客户是否正在询价竞品"）。这种反转即使四关字段齐全，也
**不得置 machine_confirmed**，必须置 needs_human_falsifier_check，把"核失效条件"
这件事显式交接给人工审核（见《人工审核深化规范》增量二）。只有失效条件能被已有
证据/检索语料直接证否的，才允许 machine_confirmed。
甬辉范例（做对的样子）：

naive_reading：前三大客户占 65%、单一大客户 35%，是危险的集中度风险，应尽快分散。
reframe：这层"依赖"是被深度绑定的工程/Builder 渠道关系，是别人进不来的壁垒。
mechanism：进该渠道需要特定认证 + 多年合作信任 + 定制化适配，三者叠加形成高切换成本，新进入者难以撬动。
falsifier：若该大客户正在询价竞品、或所需认证已被同行普遍获得、或合同是短期可随时切换的——则反转不成立，65% 就是纯粹的风险。
depends_on：认证/独有能力清单（Plus 项）。没有这个 Plus 数据，机制无法证实，反转应降级为"待验证"或不出。 这与《采集清单》"反转强依赖认证/独有能力清单 Plus 项"严格咬合。


由此引出对"每维必有反转/想不到"这条标准的一处调整建议（请你定夺）：把"想不到发现"的硬指标从逐维改为报告级。逐维提示词追求洞察、但允许诚实地说"这一维符合预期（strength=low）"；由综合环节（Step 3）保证整份报告至少有 2–3 个过关的强发现（正好喂给"三个关键发现"页）。这样既守住制作规范的洞察底线（报告级有保证），又避免逐维硬凑制造的假洞察。逻辑闭环不变，只是把质量门从"每一维"挪到"整份报告"。


四、检索与取数机制（治"模型脑补外部数据"病）

决策备忘的红线：外部信息必须实时检索、带源，禁用模型记忆。这同样不能靠在 prompt 里写"请只用检索到的数据"实现——模型会一边答应一边脑补。必须用"先检索、后推理"的两步结构从机制上锁死。

两步结构（市场、竞争两维强制）：
Step A · 检索调用：针对该维需要的外部事实（市场规模、增速、政策、对手公开数据、
行业健康线…）发起真实检索，产出"来源语料"——一个 list，每条：
  { "claim": "事实陈述", "value": "数值/区间", "source_url": "...",
    "source_tier": "一手权威 | 可信二手 | 仅线索" }

Step B · 分析调用：提示词里只注入这份来源语料作为"唯一合法外部事实库"，并加铁律：
  - 任何外部数字，只能引用语料中的条目，并在 evidence 里带上 source_url。
  - 语料里没有的外部数字，一律不得出现；可写"未检索到，留待补充"。
  - source_tier="仅线索"的内容只能用于发现痛点方向，不得作为论据数字。
对手信息的特别处理（决策备忘第 7.3 节落地）：
对每条竞争对手信息，标 source_type：
  - "verified"：上市对手财报/公开产品价格渠道——带源用。
  - "inferred"：私有公司内部数据查不到——不要编，改用"行业典型水平/可观察的
     市场行为"推断，并在 mechanism 里写明这是推断及推断依据。
宁可写"只能基于公开信息推断"，绝不给看似精确实则编造的对手数字。

把 source_type / source_tier 做进输出 schema，是为了让报告生成模块能在视觉上区分"硬数据"与"判断"（决策备忘 7.2 手法③）——硬数据带源标记，判断标推理依据。


五、财务维度的特殊契约：解读，不计算

财务数字由确定性 Python 模块算出（-410 万、65%、1.6 个月、1892 万…）。财务维的大模型调用绝对不准重算、不准"调整"、不准生成任何新数字。它的唯一职责是解读已算好的数字。

你会收到一份 financial_facts（来自财务计算模块的确定性结果），其中每个数字都是
ground truth。你的任务不是计算，是解读：
  - 严禁产出任何 financial_facts 之外的新财务数字。
  - 严禁修改、四舍五入、重新推导其中任一数字。
  - 你做的是：指出哪个数字最致命、它意味着什么（走 So-What 链）、它在五维里
    还会从哪个角度被重新照射（如客户集中度同时是竞争资产）。
若解读需要一个 financial_facts 里没有的数字，写"需补充计算：XX"，不要自己编。
财务维输入契约示例（财务模块 → 财务维，完整形态，Plus 财务全部齐备）：
json{
  "tier": "full",
  "product_lines": [{"name":"法兰/排水配件","full_cost_net":-410,"revenue":4200,
                     "revenue_share":0.226,"unit":"万元","is_loss":true}, ...],
  "customer_concentration": {"top3_pct":65, "top1_pct":35, "benchmark_healthy":40},
  "cash_runway_months": 1.6,
  "ar": {"balance":4960, "days":97, "releasable_at_60days":1892, "unit":"万元"}
}

revenue_share（每条线营收占比）是必备字段——第九节"亏损线占营收>15%"的评分硬规则要用它（裂缝修补 V1.1）。

残缺形态（关键）：财务模块的输入横跨基础项（账上现金、月度刚性支出→只够算
cash_runway_months）和 Plus 项（产品线明细、客户占比、应收→才能算出血点/集中度/可释放）。
客户只填基础时，financial_facts 是残缺的，财务维必须显式处理，不得假装数据齐全：
json{
  "tier": "basic_only",
  "product_lines": null,            // 无法识别隐性出血点
  "customer_concentration": null,   // 集中度只能转交竞争维定性处理
  "cash_runway_months": 1.6,        // 仍可算
  "ar": null                        // 无法给可释放测算
}
财务维 prompt 追加铁律：
先读 financial_facts.tier。若为 basic_only：
  - 只对 cash_runway_months 做硬判断；
  - product_lines/customer_concentration/ar 为 null 的部分，降级为定性提示
    （"现金跑道偏紧；若提供产品线与应收明细，可精确定位出血点与可释放金额"），
    置 degradation.degraded=true、列出对应 missing_plus 与 upgrade_hook；
  - 严禁为 null 字段脑补任何数字。

六、通用系统提示词（所有五维共用，注入每次调用）
你是「泽思 NBG 五维诊断分析引擎」，不是通用助手。你为一家具体企业做某一个维度的
增长诊断，最终产出给企业老板看。遵守以下铁律：

【角色与立场】
- 结论先行：先抛出一个有棱角的判断，再给支撑（金字塔原理）。
- 客户视角、第二人称"你"：把抽象命题翻译成客户自己的处境和后果，让他一读就知道
  "这在说我"。锚定客户真实经验，不用分析框架视角。
- 去口语化，但不去棱角：判断鲜明有力，用书面、概念化语言承载；
  禁用"红海/卷/握牌/唠"这类自媒体腔。
  ✗ "你在一片红海里和所有人拼价格"
  ✓ "你把最大的力气，花在了最不赚钱的战场上"
  ✗ "竞争结构判定：正面价格竞争不具备可持续性"（框架视角，冰冷）
  ✓ "你越靠压价守单，毛利就越薄——这条路走不通"（客户视角，有代入感）

【对发现慷慨，对处方克制】
- 诊断只点到"问题是什么、有多严重、为什么"，给"方向"。
- 不给"具体怎么做/路径/操作"——那是方案阶段的内容，本环节越界即扣分。

【数据红线】
- 外部数字只能引用提供给你的来源语料，带 source_url；语料没有的不得出现。
- 财务数字是 ground truth，不得新增、不得修改。
- 对手私有数据查不到就标 inferred 并说明推断依据，绝不编造精确数字。

【输出】
- 只输出符合指定 schema 的严格 JSON，不要任何前言、解释、Markdown 代码块标记。

七、逐维 Prompt 规范

每维统一输出下面的信封 schema（字段固定，便于报告生成模块消费）。各维只在「框架镜头/要逼出的核心判断/反转倾向」上不同。

统一输出 schema：
json{
  "dimension": "market|competition|business_model|capability|finance",

7.1 维度一 · 市场与机会

框架镜头：MECE 机会拆解 + 波特行业分析。
核心问题：这个行业还有没有增长空间？机会在哪个方向、窗口期多久？
要逼出的核心判断：不是"行业还行"，而是用 MECE 拆出客户没在看的那个机会方向，并判断窗口期。
输入：行业子品类、主力市场、想拓展方向（基础）；客户感知的需求变化（Plus）；来源语料（必需，走第四节检索）。
专用提示词要点：

用 MECE 把该子品类的增长机会拆成相互独立、完全穷尽的几支（按规模排序），
基于来源语料判断每支的规模与窗口期。重点找"客户只盯着最显眼那支、却忽略了
旁边更适合自己的那支"。所有市场规模/增速数字必须来自来源语料带源。
窗口期是"判断"不是"事实"，须讲清推理依据，不伪装成硬数据。

降级：未填拓展方向 → 仅给"基于行业的机会判断"。需求变化缺失 → 仅用检索数据，不结合客户主观。

7.2 维度二 · 竞争格局

框架镜头：波特五力 + 蓝海客户价值曲线。
核心问题：你和对手比，结构性优势与劣势在哪？
要逼出的核心判断：竞争定位 + 差异化空白点（别人没做但客户需要的）。反转最易出现在此维。
输入：主要对手（基础）、客户最看重什么（基础）；各竞争维自评（Plus，画"你"的价值曲线）、资质/认证/独有能力清单（Plus，反转的关键依赖）；来源语料（对手公开信息）。
专用提示词要点：

波特五力判断"这个仗好不好打"（行业层竞争强度），客户价值曲线判断"你打得怎么样"
（企业层位置）。两者叠加，直接暴露差异化空白点。
对手数据严守 verified/inferred 区分（第四节）。
尝试反转：客户视为软肋（如大客户依赖）的东西，是否其实是高切换成本壁垒？
按 3.3 四关检验，机制依赖"认证/独有能力清单"Plus 项——缺该项则反转降级为待验证。

降级：对手缺 → 用检索的典型竞争格局替代。自评缺 → 价值曲线只画行业线、不画"你"线。能力清单缺 → 只给一般性竞争判断、不出反转。

7.3 维度三 · 商业模式

框架镜头：商业模式画布。
核心问题：你的赚钱逻辑可持续吗？哪个环节最脆弱？
要逼出的核心判断：模式健康度 + 最脆弱环节（往往出在"模块之间的匹配关系"，不是单个模块）。
输入：收入主要来自哪些产品/服务、靠什么赚钱/留客（基础）；收入结构占比（Plus）；财务 ground truth（共享，判断脆弱性需要毛利真相）。
专用提示词要点：

用九模块画布拆赚钱逻辑。重点不是逐一描述模块，而是审视模块间的匹配：
价值主张是否还独特？关键资源是否正被替代？收入结构是否过度依赖某一环？
结合财务 ground truth：若某产品线负毛利（如法兰线年亏410万），说明"成本结构"
与"价值主张"之间已出现结构性错配——这是脆弱环节的硬证据，不是猜测。

降级：收入占比缺 → 脆弱环节判断偏定性。

7.4 维度四 · 内部能力

框架镜头：能力-资源矩阵。
核心问题：现有团队和资源，能不能支撑你想要的增长？
要逼出的核心判断：能力匹配度 + 关键短板（哪些必须补、哪些可外包）。
输入：团队结构、关键职能自评强弱（基础，颗粒度对齐竞争维价值曲线）；数字化水平、关键人才依赖（Plus）。
专用提示词要点：

把现有能力/资源与"增长所需"做对比，找差距。注意：增长所需的能力，应来自
市场维(机会方向)和竞争维(差异化空白点)隐含的要求——但本调用不直接读它们的结论，
只基于客观能力自评给出本维判断，跨维匹配留给综合环节。
短板按"差距大小×对增长的关键性"排序，标"自建/外包/找伙伴"倾向（只给方向不给细案）。

降级：数字化/人才依赖缺 → 矩阵颗粒度变粗。

7.5 维度五 · 财务健康度

框架镜头：杜邦分析 + 全成本作业法 + 营运资金周期模型。
核心问题：你的钱撑不撑得到转型成功那一天？
要逼出的核心判断：财务风险等级 + 隐性出血点（客户模糊知道、但从未量化的问题）+ 现金安全边际。
输入：financial_facts（财务模块 ground truth，第五节契约）。
专用提示词要点：见第五节"解读不计算"契约。核心是把已算出的数字（如法兰线 -410 万、集中度 65%、跑道 1.6 月、应收可释放 1892 万）走 So-What 链，逼出"你以为它保量，实际每年悄悄亏 410 万"这类一量化就清晰的结论。
评分特别规则：见第八节——财务维评分应部分由硬指标规则锚定（如现金跑道 < 2 个月，本维评分上限封顶），不全交模型自由判断。


八、交叉验证与综合（Step 3，最值钱的一环）

这一环把五份独立输出，变成"单点提问给不出"的全局洞察。它不是汇总，是重新照射 + 筛选 + 定调。用最强模型。

综合提示词四项任务：
你会收到五维各自的诊断 JSON。完成下列任务，输出 synthesis schema：

0) 汇编 findings[]（finding_id 的唯一户口，最先做）：把五维里所有"可被开方"的发现
   （核心判断、关键证据、确认的反转）收成一个规范数组，每条分配一个全局稳定 id
   （F01、F02…）。这是 finding_id 的唯一来源——三大关键发现、确认反转、以及下游
   方案的 traces_to，全部引用这里的 id，任何环节不得另造 id。
   每条：{id, dimension, statement, type:judgment|evidence|reversal, strength, quant_value}

1) 交叉呼应：找出"同一事实在不同维度有不同甚至相反含义"的点，至少找1组。
   范例：客户集中度65%——在财务维是风险，在竞争维是壁垒资产。这种交叉是NBG的护城河。

2) 反转筛选：审视五维提交的所有 reversal_candidate，按 3.3 四关复核，
   剔除不可证伪/无机制/不改决策的伪反转，保留过关的（可结合交叉呼应升级其说服力）。
   ★ 透传每个反转的 status：失效条件需私有/默会信息核验的，保持
     needs_human_falsifier_check，不得擅自升为 machine_confirmed（交人工审核）。

3) 总体判断 + 评分一致性：给一句话总体判断（须具体、点出该企业真实结构性问题，
   不能是"亚健康/需干预"这种笼统标签）。复核五维评分是否自洽，给总分(五维等权平均)。
   范例(具体)："不是不行，而是'外强中干'——市场竞争尚可，越往里的模式、财务越弱，
   最紧迫的是先止住财务失血。"

4) 三个关键发现：从 findings[] 里选出最有杀伤力的2-3个（会被截图转发的那种），
   用 finding_id 引用，保证整份报告至少有这几个"客户想不到"的强发现（报告级洞察门）。

5) 一致性自检：检查五维结论之间有无矛盾（如财务说紧张、但某维暗示大笔投入）。
   有矛盾则标出，留待人工审核。
synthesis 输出 schema：

九、评分规则（喂体检评分页，不可让模型自由飘）

制作规范要求一页醒目的"健康度体检表"。评分若全由模型自由给，会在不同企业间漂移、不可比。规则锚定 + 模型微调：


评分区间锚定：8–10 健康 / 5–7 亚健康 / 1–4 警告（与方法论一致）。
模型给分时必须在 rubric_basis 写明对照的判据，不准只给一个数。
财务维评分部分由硬规则封顶（防漂移），规则直接读 financial_facts 字段，例如：

cash_runway_months < 2 → 财务维评分 ≤ 3（警告）。
任一 product_lines[].is_loss=true 且其 revenue_share > 0.15 → 评分 ≤ 4。
customer_concentration.top3_pct > 60 → 评分 ≤ 5。
（tier=basic_only 时，后两条因数据缺失不触发；财务维评分只受 cash_runway_months
规则约束，其余由模型在区间内定性微调，并按降级标注说明"数据补齐后可重新评分"。）
（上述阈值由你和顾问定标，写进财务模块或综合环节的规则层，模型只在规则允许的区间内微调。）


总分 = 五维等权平均（方法论既定）。雷达图形状比单分重要——综合环节须点出"短板决定上限"。


十、降级与升级钩子（契合两价位档商业逻辑）

任何 Plus 缺失：不报错、不阻断，该维降级为定性判断，degradation.degraded=true，列出 missing_plus 和 upgrade_hook。
upgrade_hook 是给报告生成模块的素材，用于在该维标注"提供 XX 数据可解锁精确量化"——既诚实，又自然引导升级到 ¥1999 档。
基础档报告必须仍有洞察力：市场/商业模式/内部能力三维基础项即可出彩；竞争、财务两维的最锋利发现（反转、量化出血点）依赖 Plus——这正是两个价位档的真实价值差，提示词不得用套话填补这个差。


十一、红线自检（编排管线在输出前自动跑一遍）
报告生成前，对综合结果跑一遍机器可检的自检，任一不过则打回/标人工审核：

 每个外部数字都有 source 且 source_type≠model_memory（无脑补数字）。
 财务数字全部能在 financial_facts 里找到原值（模型没新增/改动数字）。
 每个关键数字都带 benchmark（无裸数字）。
 每个 confirmed_reversal 四关字段齐全且 confidence≥0.6（无伪反转）。
 每维 core_judgment 是观点结论句、非主题标签（"你的大客户依赖是壁垒不是软肋" ✓ / "竞争分析" ✗）。
 全报告 ≥2 个 strength=high 的发现（报告级洞察门）。
 无越界处方（诊断里没出现具体执行路径/操作步骤）。
 consistency_flags 为空，或已标记人工审核。


十二、接口约定（与相邻模块的数据契约）

上游输入：采集系统按《采集清单》schema 给结构化 JSON；财务模块给 financial_facts（第五节契约）。
本模块产出：五维 JSON 数组 + synthesis JSON。
下游消费：报告生成模块读上述结构，按《制作规范》套版式——core_judgment→action title；evidence 带 source_type → 区分硬数据/判断的视觉；score→体检评分页；three_key_findings→关键发现页；cross_resonances→交叉呼应的全局洞察；transition_to_solution→收口页。
人工审核接口：决策备忘定义的"人工审核深化"在本模块产出后介入——顾问读 synthesis 的 consistency_flags 和各维 strength=low 项，补行业洞察、深化关键发现。这是最值钱的环节，提示词不替代它，只为它准备好高质量初稿。
